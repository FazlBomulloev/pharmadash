## 2. Неиспользуемые поля

### Быстрые win'ы (1–3 часа работы каждый)

1. **ATC-класс** → новый KPI «Терапевтический класс» в Zone1, позволяет сравнивать рынки внутри одного класса (бенчмаркинг).
2. **БГ vs Г share** → ringchart рядом с RET/HOS, показывает зрелость рынка (генерик-доля).
3. **Top-ТМ (бренд-уровень)** → таблица под «Топ-10 корпораций» — конкуренция бывает на уровне ТМ, а не только корпорации.
4. **`pack_size` парсинг** → распределение упаковок (мелкие/средние/крупные); влияет на канал сбыта.
5. **ГРЛС reg_date гистограмма** → bar по годам регистраций (видно когда рынок начал «насыщаться»).
6. **ГРЛС expire_date предупреждения** → «через 1/2/3 года истекает N РУ» — окна возможностей.
7. **Цена за единицу из ПЦ** = `price_rub_no_vat / pack_qty_parsed` → распределение цен (boxplot/histogram).

## 3. БДП-расширения

### 3.1 Жизненный цикл рынка (LCM-фаза)

**Что:** Классификация — Introduction / Growth / Maturity / Decline.

**Формула:**
```
если usd_growth_avg_3y > 15% и un_growth > 10%: Growth
если abs(usd_growth) < 5% и un_growth близок к 0: Maturity
если usd_growth < -5%: Decline
иначе: Stable
```
Текущий `_classify_market_status` использует только 1 год — расширить до 2–3 CAGR.

**Зачем:** BD не заходит на Decline-рынки без специальной стратегии (последний игрок).

### 3.2 Концентрация по сектору (раздельно)

Сейчас HHI/top-3 общие по МНН. **Полезнее:** HHI отдельно для RET и для HOS — рынки часто структурно разные (госпиталь — олигополия, розница — фрагмент).

**Формула:** уже есть в `_build_zone2`, нужно разделить `producer_data` по сектору и считать HHI дважды.

### 3.3 Энтропия рынка (Shannon)

```
H = -Σ(s_i × log2(s_i))    где s_i — доля производителя
```

`H=0` — монополия, `H=log2(N)` — равномерное распределение. Дополняет HHI: HHI сильно реагирует на лидера, энтропия — на «равность» хвоста.

### 3.4 Динамика конкурентов (winners/losers Y2 → Y3)

Для каждого производителя посчитать `delta_usd = usd_y3 - usd_y2` и `delta_share = share_y3 - share_y2`.

Виджет «Кто отнимает долю»: топ-3 растущих + топ-3 падающих.

### 3.5 Региональная концентрация (Gini)

```
Gini = (2 × Σ(i × x_i)) / (N × Σx_i) − (N+1)/N
```
По продажам в регионах для конкретного МНН. Если Gini > 0.7 → рынок «локализован», стратегия должна это учесть.

### 3.6 Эластичность спроса (грубая оценка)

```
elasticity ≈ (un_growth) / (asp_growth)     если asp_growth ≠ 0
```
Знак: эластичный (`|e| > 1`) или неэластичный (`< 1`). Понимание — насколько UN падает при росте цены.

### 3.7 ATC-сравнение (терапевтический класс)

Среди всех МНН того же ATC-3 класса посчитать медиану USD/UN/HHI → показать «где наш МНН относительно соседей по классу».

### 3.8 Доля «оригинал vs дженерики» (упрощённо)

Из `bg_g` уже есть. Добавить долю продаж по двум категориям + средний ASP БГ vs Г → видно ценовой gap.

## 4. ГРЛС и ЖНВЛП

### 4.1 Дата регистрации — гистограмма входа конкурентов

`reg_date` → группировка по году → bar-chart «новые РУ в год».
Тренд показывает фазу регуляторной активности (насыщение / открытие).

### 4.2 Возраст рынка

`market_age = current_year - min(reg_date)` — сколько лет первой РУ существует.
Молодой рынок (<3 лет) — поздно зайти если уже плотно; зрелый — наоборот, освобождение слотов.

### 4.3 Carrousel истечения РУ

```
window_1y = count(expire_date в [today, today+1y])
window_3y = count(expire_date в [today, today+3y])
```
Карточка: «В ближайшие 12 месяцев истекает N из M активных РУ».
**Зачем:** освобождение слотов на конкретном рынке — окно входа.

### 4.4 Российские vs иностранные регистранты

По `ru_holder_canonical` (можно по эвристике — наличие OOO/АО/JSC/Ltd/GmbH/etc).
Доля иностранных регистрантов = риск ценовой/санкционной зависимости.

### 4.5 Регистрационная плотность по форме/дозе

`grls_per_form = group_by(lf_canonical from GRLS).count()` — для каждой формы сколько РУ.
**Видеть:** «Свечи есть в БДП = 0, в ГРЛС = 3 РУ» → значит зарегистрированы, но не продают → можно занять.

### 4.6 Зарегистрировано-но-не-продают (latent threats / opportunities)

```
GRLS_canonical_holders ∖ BDP_canonical_producers
```
Производители с активным РУ, не присутствующие в БДП. Два кейса:
- Если их **много** — рынок скоро взорвётся конкурентами (red flag для BD).
- Если их **мало** — относительно стабильно.

### 4.7 Продают-но-нет-РУ-в-нашей-базе (data quality / risk)

```
BDP_canonical_producers ∖ GRLS_canonical_holders
```
Может означать пробелы в данных ГРЛС (выгрузка устарела) или фактически — серый импорт / параллельные РУ. Diagnostic-метрика.

### 4.8 ЖНВЛП-доля по продажам

Сейчас `jnvlp_flag` — булево «хоть один РУ в ЖНВЛП». Полезнее:

```
jnvlp_sales_share = sum(usd_y3 where producer is in JNVLP-holders) / total_usd_y3
```
То есть **какая часть денег рынка** под ценовым регулированием, а не «есть ли вообще».

### 4.9 Время от регистрации до выхода на рынок

Если знаем reg_date РУ и можно сматчить с продажами — `time_to_market` median по новым игрокам. Очень BD-полезно.

## 5. Предельные цены

### 5.1 Цена за единицу (нормализация)

```python
# Парсинг pack_qty → число (например, "30 таб." → 30)
PACK_QTY_RE = r"(\d+)"
unit_price = price_rub_no_vat / parsed_qty
```

Распределение `unit_price` (boxplot) — конкретная стоимость единицы препарата на рынке. Сравнимо с ASP БДП в RUB.

### 5.2 Свободная маржа

```
asp_rub = (usd_y3 * fx_rate) / un_y3
free_margin = (asp_rub - max_pc_unit_price) / max_pc_unit_price
```

- `free_margin > 0` — рынок продаёт **выше** ПЦ потолка (странно — повод проверить данные или возможна доплата).
- `free_margin ≈ 0` — рынок упёрся в потолок (риск для маржи новичка).
- `free_margin << 0` — большой headroom (хорошо).

⚠️ Нужен курс `fx_rate` (см. §9).

### 5.3 Доля продаж под ПЦ

Сматчить `BdpRaw.tm` с `PcEntry.tm` (по `producer_canonical` × `tm_canonical` если есть). Получим:

```
pc_covered_share = sum(bdp.usd_y3 where (mnn,tm,producer) is in PC) / total
```

«Какой % рынка под ценовым контролем» — важнее чем «есть ли вообще ПЦ» (текущий `pc_flag`).

### 5.4 Свежесть индексации ПЦ

`days_since_indexation = today - max(price_effective_date)`.

Если > 24 мес — рынок «застоявшийся» в цене, скоро индексация → возможный пересчёт.

### 5.5 Лидеры по ПЦ-владельцам

`owner_canonical` group + count + sum price — кто контролирует регистрационную цену в данном МНН. Это **не всегда совпадает** с реальными продавцами (одни регистрируют, другие импортируют) — диагностично.

### 5.6 Сравнение разных pack_qty (цена за дозу)

`price_per_dose = unit_price / dose_strength` — если можно распарсить `dosage` в мг → получаем **сопоставимую** метрику между разными упаковками и дозировками.

## 6. Кросс-источники

### 6.1 БДП × ГРЛС

| Метрика | Формула / Идея |
|---|---|
| Latent competitors | `count(GRLS active holders ∉ BDP producers)` |
| Data quality alarm | `count(BDP producers ∉ GRLS holders)` |
| RU-coverage of sales | `Σ usd_y3 (производители с РУ) / total` |
| Регистрационная плотность на единицу продаж | `count(active RU) / log(usd_y3)` |
| ЖНВЛП-доля денег | `Σ usd_y3 (производители в ЖНВЛП) / total` |

### 6.2 БДП × ПЦ

| Метрика | Формула |
|---|---|
| ПЦ-покрытие продаж | `Σ usd_y3 (под ПЦ) / total` |
| ASP vs PC scatter | каждый конкурент — точка (ASP, PC_ceil) |
| Конкуренты у потолка | `count(производители где ASP/PC > 0.85)` |
| Headroom для нового игрока | `median(PC_unit) - median(ASP_unit)` |

### 6.3 ГРЛС × ПЦ

| Метрика | Идея |
|---|---|
| Возрастной разрыв | `median(reg_date) − median(price_reg_date)` (РУ молодые, а цены старые? или наоборот) |
| ЖНВЛП-ПЦ корреляция | сколько % РУ из ЖНВЛП реально имеют ПЦ |

### 6.4 БДП × ГРЛС × ПЦ — комплексные сценарии

```
сценарий = (есть_РУ, есть_продажи, есть_ПЦ)
```

| Кейс | Смысл |
|---|---|
| (РУ ✅, продажи ✅, ПЦ ✅) | стандарт, регулируемый рынок |
| (РУ ✅, продажи ✅, ПЦ ❌) | свободное ценообразование |
| (РУ ✅, продажи ❌, ПЦ ❌) | **latent — можно зайти быстро** |
| (РУ ❌, продажи ✅, ПЦ ❌) | data quality / серая импорт |
| (РУ ✅, продажи ❌, ПЦ ✅) | latent + ценовое ограничение |
| (РУ ❌, продажи ❌, ПЦ ❌) | дикое поле — никого нет (странно для МНН) |

Эта таблица — основа для **Opportunity Matrix** виджета.

## 8. Расширения Decision Engine

### 8.1 Новый блок «Барьеры входа» (15 баллов от регуляторики)

Добавить в `calculate_regulatory_score`:

```python
# Latent threat score
if latent_competitors_count > 5: -3  # рынок насыщен скрытыми конкурентами
elif latent_competitors_count == 0 и grls_active_count < 3: +3  # пусто

# Expiry window
if expiring_1y_share > 0.3: +4  # окно входа большое
elif expiring_3y_share > 0.5: +2

# JNVLP money share (а не флаг!)
if jnvlp_sales_share > 0.7: -3
elif jnvlp_sales_share > 0.3: -1
else: +2
```

### 8.2 Экономический скор — добавить:

- **CAGR 2y вместо 1y growth** — устойчивее, не реагирует на разовый bump.
- **ASP volatility** — std(asp_y1, y2, y3) → ценовая нестабильность как риск.
- **Sales per registrant** = `usd_y3 / grls_registrants` — «эффективность игроков». Низкая = рынок забит малопродающими игроками.

### 8.3 Структурный скор — добавить:

- **Раздельный HHI RET/HOS** — каждый по 3 балла (вместо общих 5).
- **Энтропия** — комплементарно к HHI, 2 балла.
- **Региональная концентрация** — Gini, 3 балла.

### 8.4 Новые драйверы / red flags

```
драйверы:
- "Окно входа: истекает X РУ в ближайший год"
- "Низкая регистрационная плотность (N регистрантов, < 5)"
- "Растущий рынок (CAGR 2y > 15%)"
- "Регулятивно чистый: не ЖНВЛП, нет ПЦ"

red flags:
- "ЖНВЛП-доля денег > 70%"
- "Latent threat: > 5 РУ без продаж"
- "Регистрации заморожены > 3 лет"
- "ASP volatility > 20%"
- "Концентрация HOS-сектора > 80%"
```

## 9. Парсинг — что добавить

### 9.1 Парсинг pack_qty / pack_size

В `services/normalize.py` добавить:

```python
PACK_QTY_RE = re.compile(r"(\d+(?:[.,]\d+)?)")
def parse_pack_qty(value: str | None) -> float | None:
    if not value: return None
    m = PACK_QTY_RE.search(value)
    return float(m.group(1).replace(",", ".")) if m else None
```

Применить:
- `PcEntry.pack_qty_parsed` (новая колонка `Float`) — для unit_price.

## Приложение: оценка покрытия данных по МНН

Для каждого МНН после парсинга считать «индекс полноты»:

```
completeness = w1*has_BDP + w2*has_GRLS + w3*has_PC + w4*has_ATC + w5*has_strength_normalized
```

Показывать в шапке дашборда: «Полнота данных: 4/5 ✅». Низкая полнота — пометка, что Decision Engine может ошибаться.

## TL;DR

**Главные находки:**

1. **Половина полей ГРЛС лежит мёртвым грузом** — `reg_date`, `expire_date`, `tm`, `lf_full`, `dosage`. Это самый ценный неиспользованный источник для BD-аналитики (окна истечения, тренды регистраций, latent competitors).
2. **ПЦ используется только как агрегат min/median/max** — `pack_qty`, `owner`, `tm`, `lf` дают цену за единицу, конкретных владельцев, кросс-матчинг с БДП.
3. **`pack_size` и `atc` в БДП не показываются на дашборде** — низко висящие фрукты.
4. **ЖНВЛП-флаг булев** — гораздо полезнее показывать **долю денег рынка** под ЖНВЛП.
5. **Скоринг не учитывает временные окна** — истекающие РУ, заморожённые регистрации, возраст ПЦ.

**Топ-5 быстрых улучшений** (1 спринт):
1. ATC в Zone1 + бенчмарк по классу.
2. Trend ГРЛС-регистраций по годам.
3. Карточка истекающих РУ.
4. Цена за единицу из ПЦ.
5. ЖНВЛП-доля денег (а не флаг).
